# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A pure-frontend tool that loads a **Cocos Creator 3.8.x** project (via Chrome's File System Access API, no backend) and builds an asset-usage + dependency-topology view: images, plist/Spine atlases, bitmap fonts, prefabs, scenes, and component scripts. The parsing core is DOM-free and runs identically in the browser and under Node, so it can be validated headless.

# entry-point flags: -h/--help, -v/--version, and -C <projectDir> (git-style, an alternative to the leading positional

`coir analyze [section]` is the project-wide audit — the headless `node-run.js` report exposed as CLI sections (`stats` counts/edge-kinds/metaErrors health · `unused` zero-referrer non-`resources/` assets · `orphans` dangling refs, `--dropped` adds the source-less-meta audit · `atlas` per-atlas frame utilization · `size` per-type totals, `--list` the largest files · `bundles` per-bundle size/members/degree + the **cross-bundle dependency links** (each with the contributing asset refs) + **cycles** (bundle pairs linked both ways) + **duplication** (axis D — bytes the build copies into ≥2 same-priority bundles, via `bundleDuplication`); no section = all). It reuses the pure `src/core/analyze.js` builders via `query.analyzeData`/`analyzeAll` (shared with the MCP `analyze` tool), so the CLI text and the MCP json come from one place. The `bundles` section is built over the parallel bundle graph (`bundleReport` → `buildBundleGraph`) and is the headless counterpart of the browser 報告 **跨 bundle 依賴** sub-tab.

`coir duplicates [files|configs]` finds **redundant assets to merge** (its own top-level command, NOT an analyze section, because — unlike the pure-over-scan analyze builders — it needs I/O). Two axes in `src/core/duplicates.js` (shared by the CLI `duplicates` command + the MCP `duplicates` tool): **`files`** = byte-identical source files with different UUIDs (size-bucket → fast 32-bit hash of only the size-colliders → final byte-equality verify, so a hash collision never yields a false positive; uses the new `FileProvider.bytes()`), each group flagged `mergeable:false` when the members' `importer`+meta-`userData` import settings differ (same bytes, different `.meta` → merging would change rendering); **`configs`** = structurally-identical `prefab`/`material`/`anim` (normalize the JSON past its volatile per-instance `fileId`, stable-stringify, hash, verify by exact normalized string — catches editor copy-paste duplicates whose bytes differ, which byte-hashing misses). Every group carries a suggested `canonical` (keep) + `redundant` (drop) → feed straight into `coir edit --all swap-uuid <drop> <keep>`. This is axes **A** (bytes) and **B** (semantics); axis **C** (pixels) is the cross-atlas frame/region dup below.

**CI rules** (`coir check` — the declarative gate). `src/core/rules.js` is a **pure** engine: a registry of named **checkers** keyed off the existing read-only reports (`analyze`/`bundleReport`/`duplicatesData`), so it does NO new parsing. A project's policy lives in a committed **`coir.rules.json`** (`[{ name, level?, …params }]`, or `{ rules: [...] }`); each entry enables one checker with a `level` (`error` default / `warn`). `evaluateRules(scan, rules, ctx)` returns `{ violations, errors, warns, configErrors }` (an unknown rule / bad param / thrown checker → a `config` violation). The CLI `coir check` loads the file (or `--rules <file>`, else a warn-only **default** health set so it's useful with no config), precomputes `ctx.duplicates` only when a duplicate rule is present (the one I/O input), prints each violation, and **exits 1 on any error, 2 on a config error, 0 clean** — so it gates CI. Built-in checkers: `max-meta-errors` · `no-dangling-refs` · `no-orphans` (`type?`) · `no-bundle-cycle` · `max-duplication` (`maxBytes`) · `no-duplicate-files` (`axis?`) · **`forbid-dep`** (a dependency-cruiser-style `from`/`to` **matcher** — `type`/`bundle`/`pathStartsWith`/`pathContains`/`basename`, each string-or-array, ANDed; reports the offending edge with paste-able `locSelector` locations) · **`no-cross-bundle`** (`from`/`to` bundle name(s), omit = any) · **`atlas-min-util`** (`min`). It's a built-in command (in `BUILTIN_COMMANDS`, so a plugin can't shadow it); the same engine drives the **MCP `check` tool** (inline `rules` or `rulesPath`, returns the result — no exit code). **Plugins extend it**: a `Plugin.rules: [{ name, check(scan, rule, ctx) }]` (same pure violations contract) is folded in via `collectPluginCheckers(plugins)` by both hosts (built-in names win). See `types/index.d.ts` (`RuleChecker`/`RuleViolation`).

**Cross-atlas sub-image dup** (axis C — the same art baked into more than one atlas, which whole-file dedup can never see because each atlas page is a different texture). The **spine** plugin's `spine-dup` (cross-`.atlas` regions) and the **atlas** plugin's `atlas-dup` (cross-`.plist` frames) each parse their atlas format for per-sub-image rects, group by name (corroborated by source dimensions: `likely` = dims agree, `name-only` = differ), and expose BOTH a CLI/MCP command (Tier A, no decode) AND a browser **report section** (`Plugin.reports`) whose members carry a `CropSpec` so the 報告 tab draws side-by-side thumbnails + runs a rotation-invariant 16×16 perceptual confirmation that upgrades the badge to `confirmed`/`different` (Tier B, browser-only). See `findSharedRegions`/`findSharedFrames` and the `reports` hook below.

`src/cli.js` answers "what does X depend on / who depends on X" for one asset (`info` instead dumps that asset's own record — type/uuid/ext/importer/size, in/out degrees, sub-assets, and the raw meta `userData`; the headless analogue of an editor "get asset info"). `<asset>` resolves by full path, basename, uuid, or `uuid@sub`; an ambiguous basename prints candidates and exits 2. Data goes to **stdout** (`-o json` for structured output; text is the default), so it is safe to pipe/parse. `--where` prints each usage site as a **paste-able selector** (`edge.locations` → `nodePath:Comp.prop`, the same grammar `edit` takes — see `src/core/selector.js`, shared with the browser usage popup); custom-script components are shown by class name (a compressed `__type__` decompressed). Meta-derived edges have no nodePath, so they show a non-selector form. Meta-derived edges (atlas→texture, font→texture, the Spine triple) and `extends` edges have no `locations`. `--type T[,T2]` (mirrors the browser's global type-filter bar) keeps only the chosen asset types: on the `deps`/`uses` **tree** it prunes to branches that *reach* one of those types — matching nodes **plus the intermediate hops leading to them** stay, dead branches drop, the queried root is always kept (build via `buildEdgeTree` → `pruneByType` → `renderTreeText`, leaving unfiltered output byte-identical); on `closure`/`find`/`-o json` it just filters the flat list.

`coir edit` is a separate, headless **in-place editor** for existing prefab/scene files. It never generates a file from scratch. Targets use the same selector grammar the query side displays — `nodePath:Comp.prop`, `[i]` for same-name/same-type/array disambiguation, `#N` absolute index — built from `src/core/selector.js` (the one canonical `componentName`/`locSelector`, also used by `--where` and the browser usage popup). `edit <file> tree` is a read-only **structure-discovery** op: it lists the node hierarchy + every component with a ready, disambiguated `nodePath:Type` selector (`-o json` for agents; `--with <Type>`/`--under <sel>`/`--depth N`), so a caller finds every editable point without parsing the file. `swap-uuid` (incl. `--all`, project-wide) is a minimal-diff text patch; everything else is parse → mutate the JSON array → re-serialize, with **real-delete + global `__id__` compaction** (no soft-delete cruft) and **template-by-example** for adds (clone a same-file skeleton → version-correct without branches). Nested prefab instances are detected and refused (edit them in their source prefab). Writes are atomic and **mtime-guarded** (refused if the file changed on disk since it was read — e.g. Cocos Creator saved it; `--force` overrides). Full design in **[docs/EDITING.md](docs/EDITING.md)**; the underlying format contract in **[docs/SERIALIZATION.md](docs/SERIALIZATION.md)**.

**The edit/query logic is a shared seam, not duplicated.** `src/edit/ops.js` is the pure write engine-orchestrator — `runEdit`/`runSwapAll`/`getData`/`treeData` resolve → load → mutate (via `src/edit/editPrefab.js`, the DOM-free byte-level engine) → return `{json, writes}`, with NO printing/exit; `src/seam/query.js` is the matching read seam (`depsData`/`infoData`/`findData`/`closureData`). `src/editCli.js` (CLI text rendering + value-flag coercion) and the **MCP server** (`src/mcp/`) both drive these same functions — so behaviour is identical across the CLI and MCP, only presentation differs. `src/seam/shared.js` holds the resolution/location helpers shared by both. `coir mcp` starts a hand-rolled (zero-dep) JSON-RPC/stdio **MCP server** exposing typed tools over that seam — unprefixed reads (`find`/`deps`/`tree`/`get`/…) + `edit_*` writes (a host namespaces them as `coir__<tool>`) — with an `fs.watch`-invalidated scan cache, serialized tool calls, and the mtime write-guard — see **[docs/MCP.md](docs/MCP.md)**.

The app **must** be served over a secure context (`http://localhost` or https) — the File System Access API is unavailable on `file://` and outside Chromium.

Two complementary test paths:
- `npm test` runs `node --test` over `test/*.test.js`. `test/cli.test.js` covers `src/cli.js` end-to-end (subprocess) against a **synthetic, in-temp-dir fixture** it builds itself (format-valid `.meta` + prefab/scene JSON, with a real `compressUuid`'d `__type__`) — no dependency on any on-disk project, so it is deterministic/CI-safe; `test/mcp.test.js` spawns the **MCP server** and drives it over JSON-RPC/stdio against the same kind of temp fixture. Add core unit tests as `test/*.test.js` and they are picked up automatically. (The glob excludes `test/node-run.js`, which is not a `*.test.js` file and would otherwise error on missing argv — note a bare `node --test` with no glob *does* pick it up and report one spurious failure; use `npm test`.)
- `test/node-run.js` is the **headless core validation against real projects**: it runs the real core against an on-disk project and prints summary, edge-kind histogram, unused/orphan/atlas/size reports, and a closure for one center node. Use it to verify any change to `src/core/`. Point it at any local **3.8.x** project (3.8.5 is format-identical to 3.8.6); 2.4.6 `.fire`-format projects will not parse. A healthy scan reports `metaErrors=0` — a nonzero count is a regression.

```bash
node test/node-run.js ../my-cocos-project scene/Main.scene
```

## Architecture

The single most important design fact: parsing is decoupled from the environment through a **`FileProvider`** interface — `listFiles()`, `readText(path)`, `size(path)`, with all paths POSIX-relative to the project's `assets/` root. There are two implementations and the core depends only on the interface:

- `src/core/` — DOM-free, no I/O of its own. Shared by browser and Node.
- `src/browser/fsapi.js` — wraps a File System Access directory handle into a FileProvider (auto-detects whether the user picked the project root or `assets/` directly).
- `src/node/fsProvider.js` — wraps `node:fs` into a FileProvider; shared by `test/node-run.js` (full report) and `src/cli.js` (dependency query).

Note `src/cli.js` reads `edge.locations` for `--where`; the `graph.js` adjacency intentionally drops `locations`, so the CLI indexes `scan.edges` directly for neighbour lookups (and reuses `closureReport`/`buildAdjacency` for the transitive `closure` command).

Entry/data flow: `app.js` (or `node-run.js`) picks a directory → builds a provider → `scanProject(provider)` → attaches `buildAdjacency(edges)` as `scan.adjacency` → hands the scan to `initUI`/the reporter. The scan result object (`{ assets, byPath, edges, subOwner, subUsage, orphanRefs, metaErrors, missing, missingReferenced, files, rootTypes, bundles }`, assets keyed by uuid; `missing` = uuid→path of dropped source-less metas, `missingReferenced` = the subset of those paths something still points at; `rootTypes` = plugin-declared never-unused types; `bundles` = the Asset Bundle list, see below) is the contract between core and presentation. The 報告 tab surfaces the dropped metas in a "缺來源檔的 meta（已略過）" section (`droppedMetaReport`), each flagged 仍被引用 (a broken dependency to fix) vs 無人引用 (a stray meta). The 報告 tab is organized as **sub-tabs** (one per section, a sticky `.rtabs` bar; `reports.js` `selectReportTab` swaps the body without re-running the report) — core sections 未使用/孤兒參照/圖集利用率/資源體積/缺來源檔meta, plus one tab per plugin `reports` section (spine/atlas cross-atlas dup). A floating **back-to-top** button (`#toTop`, `smoothScrollTop`) appears on 清單/報告 once scrolled.

**體積圖** (`sizemap.js`) is now its **own top-level tab** (between 拓撲 and 報告) sharing the topology centre `S.treeRoot` — a squarified-treemap (`treemap.js` `squarify`, unit-tested) of asset bytes scoped to the centre's dependency **closure** (à la Unreal's Size Map: "what makes this thing heavy"; root's own file included since `dependencyClosure` strips it), or the whole project when there's no centre. Outer cells = types **or bundles** (a `#smGroup` toggle, `S.sizemapGroup` — shown only when the project has real bundles; a bundle centre forces type grouping), inner = assets coloured by their own type, area ∝ size, thin-gap cells; each cell gets an **outlined filename sized to fit it** (`nameLabel` — tiny on a small cell, so a zoom magnifies it into readability), and a type's long tail lumps into a clearly-labelled `其他 N 項` tile. A bundle centre shows that bundle's **members** (`sizeMapBody` special-cases it). Assets the build **duplicates across bundles** (axis D, `S.bundleDupMap` from `bundleDuplication`) get a **red `tmdup` outline** + a `×N copies, wastes Y` tooltip. It carries the **topo-style bar** (`#smbar`): a node **filter** (left, hides non-matching cells, debounced), the group toggle, the **體積 total** (`sizemap.total`), and a **breadcrumb** (`整個專案 ‹ centre`) + copy/link. **Gestures:** single-click a cell = **drill** (re-centre via `setCenter`, stay here) — a 210ms timer lets a **double-click** pre-empt it to **jump to 拓撲** (`focus`); double-click on empty space resets the zoom; image cells `data-img` get a downscaled `<image>` thumbnail (`hydrateSizeMap`, gen-guarded); `attachSizemapZoom(svg, onPick, onJump)` does pinch-zoom + scroll-pan-when-zoomed + drag-pan. **Keyboard:** arrows move a cell cursor to the spatially-nearest cell (`sizemapMove`), `Enter` drills (`sizemapEnter`), `−`/`+` step the shared centre history. The centre/selection/nav-history is unified across 清單/拓撲/體積圖: `topo.js` exports `setCenter` (re-centre + history, no tab switch) and `focus` (= `setCenter` + jump to 拓撲); `−`/`+` (`navBack`/`navForward` in `ui.js`) re-render whichever view is active.

**Viewer mode + embedding.** Besides the directory-pick flow, `app.js` checks the URL hash: a `#topo=<blob>` opens a **topology snapshot** with **no File System Access** (so it works on any modern browser, not just Chromium). `decodeTopo` (`src/core/topohash.js`) rebuilds a scan-like object from a compact neighbourhood subgraph (integer-indexed nodes, interned types/kinds, gzip+base64url) and `setScan(s, name, { viewer:true, center })` keeps the **清單 + 拓撲 + 體積圖** tabs (only 報告 + the directory picker are hidden via `body.viewer`; `cycleTab` skips 報告 too — a snapshot has no project-wide reports; the 清單 lists the snapshot's nodes, a row click re-centres; boundary nodes — real neighbours trimmed — marked `⋯`; edges whose usage detail was omitted show a generic hint). Each snapshot node tuple is `[path, typeIdx, bytes, boundary(0/1), bundleIdx]` (with an interned `bd[]` bundle-name table) — the per-node **size** lets the viewer draw the **體積圖** off the snapshot, and the per-node **bundle** drives the 清單 Bundle column + size-map group-by-bundle in the viewer (the viewer does NOT build the bundle graph — a snapshot is a partial neighbourhood — it only carries the field). NOTE both bytes and bundle were **breaking** payload changes (old `#topo=` links won't decode), kept at `v:1`. `encodeTopo(scan, centerKey, { maxDepth=5, locDepth=2, cap })` builds the blob, auto-shrinking depth to fit `MAX_BLOB_CHARS` (256KB, tunable) and always returning a link; it runs in Node too (`CompressionStream`/`btoa` are globals — with bare `zlib` + `globalThis.Buffer` base64 fallbacks for old Node, e.g. Cocos 3.5's Electron which lacks both; the gzip is interoperable either way), so one encoder serves an embedder. `package.json` `exports` (`.` → `src/index.js`) gives a Node host `import('coir')` → `scanProject`/`buildAdjacency`/`encodeTopo`/`decodeTopo`/`makeFsProvider`/`PLUGINS`/`dedupePlugins`/`loadConfigPlugins`/`loadPluginFiles`/`COIR_ROOT` (the repo root, so a host can load the repo-root global `coir.plugins.mjs`). **`cocos-extension/`** is a Cocos Creator **3.5–3.8** editor extension built on this (`editor: >=3.5.0`; the zlib/Buffer fallbacks cover 3.5's old Node): right-click an asset → a submenu listing dependencies (→) then dependents (←) as **indented trees** (depth shown by indent — NBSP per level so the editor doesn't collapse it — each deeper layer nested under its parent, no per-node cap, no "層N" labels; each row jumps via `Editor.Selection.select`) + opens the snapshot in the browser; it runs coir-core in-process (cached scan, invalidated on asset-db change) **loading the project + coir-root `coir.plugins.mjs`** (`dedupePlugins([...PLUGINS, ...loadConfigPlugins(COIR_ROOT), ...loadConfigPlugins(projectPath)])`, logging the active ones as `source.name` — so editor-side edges match the CLI/browser, e.g. audio-call), and `install.sh` copies it into a project's `extensions/` + symlinks `node_modules/coir`. NOTE: the asset menu builder MUST be **synchronous** (the editor doesn't await an async one → the menu silently vanishes), so degree/graph data is pushed to it via `request` + `broadcast`. The same pump carries plugin **`assetMenus`** (e.g. `anim`/`skel` from coir-plugins): `main.js` runs each `rows(ctx)` over matching assets in the background (mtime-cached, so an unchanged `.skel` isn't re-parsed) and pushes a `uuid → [{label, rows}]` map; the sync builder renders a single row flat (`Coir anim  0.33s`) and multiple rows as a submenu. Since Node caches the ESM `import`, adding/editing a `coir.plugins.mjs` plugin needs a full **editor restart** (an extension reload doesn't clear the module cache); `install.sh` copies the extension's own files, so changing them needs a re-install. The extension also ships a dockable **`Coir 跳轉` (goto) panel** (`panels/goto.js`; menu + `Ctrl+Alt+G`): type a coir `nodePath`(+`[i]`) → select that node in the **live scene** (`scene` `query-node-tree`), or a `xxx.prefab`/`ui/foo.png` filename → select that **asset**; **reverse** — selecting a node/asset backfills the input with its nodePath/filename (not while typing). It strips a trailing `:Comp`/`.prop`, resolves a no-`[i]` same-name sibling to `[0]`, and doesn't support `#N` (no live-scene equivalent).

### The scan pipeline (`src/core/scan.js`) — read this first

`scanProject` runs phases in a deliberate order; the ordering is load-bearing:

1. **Parse every `.meta`** (`meta.js`) into asset records keyed by uuid. `importer` → normalized `type` via a lookup table; `subMetas` flatten (recursively) into `subAssets`, each addressable as `uuid@subId`. Directory metas (`importer === 'directory'`) are dropped — they are not assets — **but a bundle folder's directory meta is mined for its Asset Bundle config first** (`userData.isBundle` → `{ name: bundleName || folder, root, priority }`). **Source-less metas are also dropped**: a `.meta` whose source file is gone (only the stale meta lingers, `!hasSource`) is not indexed, but its `uuid → path` (plus sub-uuids) is remembered in `scan.missing` so a prefab/scene still pointing at it resolves (in 3c) to a *named* missing-source orphan ref rather than a bare uuid.
   - **1b — Asset Bundle tagging.** Every asset gets `a.bundle`: its **owning bundle name** = the deepest bundle `root` that is a prefix of its path (nested bundle wins), else `'main'`. `resources` is a built-in bundle; `main` is the implicit catch-all. `scan.bundles` is the descriptor list (`{name, root, priority, builtin?}`, always incl. `main` + `resources`). This is a first-class field that **generalizes** the old hard-coded `resources/` test — but `a.inResources` stays **path-based** (so the unused policy is byte-for-byte unchanged). Consumers: the 清單 **Bundle** column, CLI/MCP `info`, and the **parallel bundle graph** below. NOTE the planned bundle-aware unused policy + cross-bundle (axis D) work in DEVELOPMENT.md §12 builds on this.

**The parallel bundle graph** (`src/core/bundleGraph.js`, browser-only nav — `buildBundleGraph(scan)` → `{ nodes, containEdges, depEdges }`). Bundles are made navigable as **first-class nodes** WITHOUT touching the asset graph: synthetic pseudo-assets (`type:'bundle'`, `virtual:true`, `hasSource:false`, key `bundle:<name>`) carrying `size` = Σ member bytes; two edge kinds — `contains` (bundle → each member) and `bundle-dep` (bundle → bundle, aggregated from the cross-bundle asset references). **The hard invariant: these never enter `scan.edges` or bump `a.in/a.out`** — else every asset would gain a phantom in-edge from its bundle and the unused report would zero out. `ui.js` `setScan` injects the nodes into `scan.assets` (so every existing `scan.assets.get` lookup works and reports skip them via `virtual`) and builds `S.bundleAdj` (contains + bundle-dep, for topology) + `S.bundleDepAdj` (bundle-only, for the list's ∑ columns). `neighborsOf` (topo.js) merges the bundle adjacency **only for bundle keys** (an asset never shows its container → no clutter). So: the 清單 lists bundles under a `bundle` type chip (size = bundle weight, in/out = bundle-dep degree), clicking one centres the 拓撲 to show its **contained assets + dependency bundles**, and the 體積圖 of a bundle centre shows its members (`sizeMapBody` special-cases a bundle scope). `buildBundleGraph` returns empty for a project with no real bundles (everything `main`), so unbundled projects are unchanged. Each `bundle-dep` edge carries `refs` (the contributing asset→asset edges); the browser keys them in `S.bundleDepRefs` so **selecting a dependency bundle in the 拓撲 opens the usage popup listing the exact cross-bundle references behind the link** (usage.js `showBundleRefs`), which is how a `folder-001 ↔ main` cycle is traced to the real prefab/material/texture files. The same data has two more surfaces: a browser 報告 **跨 bundle 依賴** sub-tab (reports.js `bundleSection` — per-link `<details>` with cycle banners, each ref click-to-focus) and the headless **`coir analyze bundles`** / MCP `analyze {section:'bundles'}` (the pure `bundleReport`).
2. **File sizes** for assets that have a source file.
3. **Collect references RAW, then resolve.** Prefab/scene/anim/mtl JSON is walked once (`walkJsonRefs`) into `rawRefs` (`__uuid__`) and `rawTypes` (`__type__`) **without resolving them yet**. This matters because step 3b mutates the asset index.
   - **3b — component-script pruning** (the subtle part, see below) deletes non-component `.ts` assets from the index.
   - **3c** resolves the raw refs *after* pruning, so edges never point at a script that was just removed.
   - **3d — plugin-contributed edges.** The meta/convention-derived edge kinds — atlas→texture (`subAsset.userData.imageUuidOrDatabaseUri`), font→texture (`meta.userData.textureUuid`), particle→texture (`meta.userData.spriteFrameUuid`/`textureUuid`, routed through `resolveUuid` so a missing target is flagged as an orphan), and the Spine triple (skeleton `.json` → `.atlas` → page `.png`s, multi-page PNGs parsed out of the `.atlas` text, not guessed by basename) — are no longer inline. Each lives in a plugin under `src/core/plugins/` and runs here over a shared `ctx`. See **Plugins** below.
   - **3e — component→base-component `extends` edges** (class inheritance) stay in core, reusing the `baseName`/`definers` maps built during 3b pruning (there is no `import`/`require` edge kind).
4. **Degree bookkeeping** sets each asset's `in`/`out` counts.

### Plugins (`src/core/plugins/`)

The asset *types* that carry edge logic and the *edges* between them are pluggable. A plugin is a plain object — `{ name, importerTypes?, typeByExt?, jsonSourceExts?, rootTypes?, colors?, messages?, async edges(ctx)?, commands?, assetMenus?, reports?, rules? }` (`rules` = CI checkers for `coir check`, see "CI rules" above) — and the built-ins (`atlas`, `font`, `particle`, `spine`) each keep everything about their type in one file. `src/core/plugins/index.js` exports `BUILTIN_PLUGINS` and `PLUGINS` (built-ins + the gitignore-friendly `local.js`); `scanProject(fp, { plugins = PLUGINS })` defaults to it, so **the CLI, `node-run` and the browser all get the same set**.

- **Core fields** (read by `scan.js`/`meta.js`/`analyze.js`, DOM-free): `importerTypes`/`typeByExt` merge into the type resolver (`buildTypeResolver`, with the built-in `IMPORTER_TYPE` baseline; `knownTypes(plugins)` enumerates the vocabulary for CLI help); `jsonSourceExts` add source exts walked for refs in 3a; `rootTypes` become never-unused (`analyze.js` unions them via `scan.rootTypes`); `edges(ctx)` runs in 3d. `ctx` exposes the index (`assets`, `byPath`, `missing`/`missingByPath`/`missingReferenced`, `subOwner`, `subUsage`) and the graph primitives: `addEdge(from, to, kind, loc?, label?)` (the optional `label` is a human-readable edge description for display/search — falls back to kind+endpoints), `resolveUuid`, `noteSub`, and **`addNode({path, type, …})`** — adds a **virtual** node (a non-asset topology node: an event/notification/route with no file) that joins the graph + degrees but is `virtual:true`/`hasSource:false`, so the asset-health reports (unused/size/atlas) skip it; returns its stable key to wire edges to. Plus **`bundles`** (the `scan.bundles` descriptor list `{name, root, priority}` — resolve a bundle-relative load path against `b.root`), **`files`** (every asset-relative path the FileProvider lists — the formal entry to read any source via `readText`), `readText`, `mapLimit`, `uuid.*`, and a **read-only** `scripts` (`isComp`/`baseName`/`definers`/`text` — a convenience uuid→source map from 3b so a source-scanning plugin doesn't re-read; for arbitrary files use `files`+`readText`). A plugin imports nothing — it uses only `ctx` — so a third-party plugin needs no build step. **Recovering dynamic-load edges** (`resources.load('x')` etc., which static `__uuid__` analysis can't see) is left to a per-project plugin — coir bakes in no load heuristics; the recipe is in **[docs/DYNAMIC-EDGES.md](docs/DYNAMIC-EDGES.md)** and the ready-to-adapt `resources-load` plugin ships in the external [coir-plugins](https://github.com/aaronhg/coir-plugins) repo.
- **Presentation fields** (pure data, ignored by core): `colors` merge into the UI's `TYPE_COLOR` and `messages` into the i18n catalog (`registerMessages`), both folded in by `setScan` before the first render — so a plugin owns its type's color too.
- **Commands** (`commands?: [{ name, usage?, description?, inputSchema?, positional?, run(ctx) }]`, registry in `src/seam/pluginCommands.js`, shared by `src/cli.js` + `src/mcp/server.js`): a plugin adds an operation — e.g. `coir timeline <prefab>` — **once**, and it works in both hosts. It dispatches **after** the built-ins (deps/uses/closure/find/info/analyze/edit/mcp), which always win (a colliding `name` is ignored with a warning). `run(ctx)` **RETURNS** a result and never prints — `{ data, text? }` or `{ error, candidates? }` — so one definition serves both: the **CLI** prints `text` (or `JSON(data)` on `-o json`) and lists the command under "Plugin commands" in `--help`; if the command carries an **`inputSchema`** it is ALSO registered as an **MCP tool** (`coir mcp` — `tools/list`/`tools/call`), returning `data`. `ctx` is normalized across hosts: `env` ('cli'|'mcp'), `args` (a NAMED object — CLI positionals mapped via `positional`, else the inputSchema property order; MCP's JSON arguments), `scan`, `readText` (any source under assets/), `resolveAsset` (path/basename/uuid→uuid; **CLI** prints candidates + exits 2 on miss, **MCP** throws → a clean tool error), `edgeMaps`, `uuid.*`, `util.{base,kb}`, plus CLI-only `flags`/`argv`. So a Timeline/gantt analyzer lives in its own plugin, exposing `coir timeline` + a `timeline` MCP tool from one `run`, without forking coir. See `types/index.d.ts` (`PluginCommand`/`CommandResult`/`CommandContext`), `test/plugin-command.test.js` (CLI) and `test/plugin-mcp.test.js` (MCP).
- **Asset menus** (`assetMenus?: [{ ext?, types?, label?, rows(ctx) }]`) — a plugin contribution **independent of `commands`** (a plugin can ship only `assetMenus` — a pure "asset-menu plugin"). An editor host (the `cocos-extension/`) gives each asset matching `ext`/`type` a right-click entry whose rows `rows(ctx)` computes from the asset itself (`ctx` = `{ asset, scan, projectDir, readText }`); a **single row collapses to a flat item, multiple rows stay a submenu**. CLI/MCP ignore it. The external [`anim`/`skel` plugins](https://github.com/aaronhg/coir-plugins) ship both a command and an `assetMenus` (right-click `.anim` → flat `Coir anim  0.33s`; `.skel` → `Coir skel ▸` per-animation submenu). Contract in `types/index.d.ts` (`AssetMenu`/`AssetMenuContext`).
- **Report sections** (`reports?: [{ id, title?, async build(ctx) }]`) — a **browser-only** contribution to the 報告 tab. `build(ctx)` (`ctx = { scan, readText }`) returns pure DATA — a `VisualReport` of groups, each member optionally carrying a `CropSpec` (`{ page, x, y, w, h, rotate? }`). The host (`src/browser/visualreport.js` + `imagecrop.js`) renders each group as **side-by-side thumbnails cropped from each member's page** (2a, human compare) and, when crops are present, runs a generic **pixel-level confirmation** (a rotation-invariant 16×16 perceptual signature, normalized-L1 ≤ 0.10) that upgrades each group's badge `likely`/`name-only` → `confirmed`/`different` (2b). The plugin knows nothing about canvas; the browser knows nothing about the plugin's domain. CLI/MCP ignore it; snapshot/viewer mode skips it (no FileProvider — thumbnails need binary reads via the new `FileProvider.file()`/`bytes()`). The built-in **spine** plugin is the first user: its `spine-dup` report shows the same cross-atlas shared regions the `spine-dup` command finds, but with thumbnails. The browser composes plugin reports after the core sections via `S.pluginReportCache` (built once per scan, guarded by `S.reportGen`). Contract in `types/index.d.ts` (`ReportSection`/`VisualReport`/`VisualGroup`/`VisualMember`/`CropSpec`).
- **Registering one** (precedence: built-ins → global → project → `--plugin`/`use()`, composed by the caller then run through `dedupePlugins` so a later same-name plugin overrides an earlier one). **Built-in** (ships, works in CLI + browser): drop a file in `src/core/plugins/` and add it to `index.js` — that's the only in-repo registry now (`PLUGINS === BUILTIN_PLUGINS`; there is no `local.js`). **External, out-of-repo** (CLI/node, via `src/node/loadPlugins.js`): a `coir.plugins.mjs` config auto-loads from the **coir repo root** (global / cross-project) and from the **scanned project's root** (that project only) — both gitignored — and `--plugin <file>` loads explicit modules (repeatable). The CLI and `node-run` compose all of these onto `PLUGINS` (`loadConfigPlugins(COIR_ROOT)` + `loadConfigPlugins(projectDir)` + `loadPluginFiles`). A config module's `default` export is a plugin or an array of them; `.mjs` is recommended (always ESM regardless of the dir's `package.json`). **Browser** (no rebuild): `app.js` composes built-ins + the served coir-root `coir.plugins.mjs` (`loadGlobalPlugins`, a `webpackIgnore`'d dynamic import of the file served next to the page) + the picked project's `coir.plugins.mjs` (`loadProjectPlugins`, read via the File-System-Access directory handle and blob-imported, re-read per pick) + `window.coir.use(plugin)` runtime plugins, then `dedupePlugins`. After a scan the status line lists the active non-built-in plugins tagged `source.name` (`global`/`project`/`use`). The component-script pruning (3b) and the `extends` graph (3e) stay in core — **not** pluggable — but a plugin can read them via `ctx.scripts`.

### Two non-obvious things that are easy to break

**Component-script filtering.** A `.ts` is kept only if it is a Cocos *component*: it (a) is a class `extends Component`, (b) is referenced as a serialized `__type__`, or (c) transitively extends a component (resolved by fixpoint over the extends chain). The `EXTENDS_COMPONENT` regex deliberately requires **class context** (`\bclass\s+Name(?:<...>)?\s+extends\s+...Component`). Do not loosen it to a bare `extends Component`: a generic *constraint* like `getCacheObj<T extends Component>` in a utility module would then misclassify the util as a component. The `<[^{}]*?>` allowance also lets generic component bases/subclasses be detected. Plain util/enum/config modules must stay out of the topology.

**UUID compression** (`src/core/uuid.js`). Cocos serializes two different forms:
- `__uuid__` asset references are **full** hyphenated UUIDs, optionally `@subId` — never compressed.
- `__type__` script-component references are **compressed** 23-char (or 22-char `min`) tokens using the Cocos v2.0.10 base64 scheme. `decompressUuid` restores them; `looksCompressed` is a heuristic gate (a decompressed value is still verified against the asset index before an edge is created).

### Browser UI (`src/browser/ui.js`)

Pure DOM, no framework — split into focused modules under `src/browser/` (`state.js` = the one mutable `S` state bag + helpers; then `filterbar`/`list`/`topo`/`usage`/`palette`/`reports`/`copy`, with `ui.js` the orchestrator: `initUI`/`setTab`/`setScan`/`onKey` + event wiring). Four banner tabs over one content area: **清單** (sortable asset table — this *is* layer 0; click a row to pick the center), **拓撲** (a bidirectional, always-fully-expanded tree rendered as a **fixed 5-column sliding window** centered on the selected node's signed offset: dependents fan left as negative offsets, dependencies fan right as positive — **vertically virtualized**: fixed 30px rows, the tree is built once into `S.topo`, only the in-viewport rows are painted on scroll, and adaptive vertical padding (viewport/2 − row/2) centres the selection, so a hub with hundreds of dependents no longer floods the DOM), **報告** (the reports). `selectedKey` is the single source of truth for tree navigation — arrow keys/swipes construct keys rather than toggling an `expanded` set. `/` opens a fuzzy quick-open palette over a multi-source index (assets by path/uuid, sprite-frame names, usage sites — node-path · component.property · ClickEvent, matched on a kind-bearing `text` — and `edge` entries for location-less edges: meta/convention/plugin kinds like atlas→texture/extends/a plugin's audio-call, labelled by `edge.label` or the endpoint names), with scope sigils `@`frame `#`type `>`usage `~`edge-kind — `#` and `~` are TWO-PART (`#type query` / `~kind query`: the first token filters, the rest searches the survivors; `~` filters on the entry's `edgeKind`). Every result's `target` is an asset uuid, so picking any kind focuses an asset (`buildSearchIndex` in `palette.js`, rebuilt per scan, exported + unit-tested); the result list is **virtualized** (fixed 32px rows + click delegation) with the old 100-result cap lifted, so every match is reachable by scrolling. Global keys (`onKey`): `Tab` cycles the tabs (`Delete`/`Backspace` = reverse), `Esc` clears the type filter (after first closing the usage popup if open), `/` or `Ctrl/⌘+P` opens the quick-open, `Ctrl/⌘+C` copies the selected node's name, and (on 拓撲) `Ctrl/⌘+F` opens an in-topo find. Selecting a tree node auto-shows a "used where" popup (the location(s) of the edge between it and its tree-parent) and surfaces its full path in the status line. Per-tab keys: in **拓撲**, arrows / two-finger swipe navigate and `Enter` re-centres, `−`/`+` step back/forward through a centre+selection history (`navHistory`/`navForward`), and `Ctrl/⌘+F` opens an **in-topo find** that searches the tree's cell *data* (virtualization keeps off-screen cells out of the DOM, so the browser's native find can't see them) and scrolls to / amber-highlights matches — `Enter`/`⇧Enter` cycle, `Esc` closes — the find is a **floating overlay** at the tree region's top-right, a sibling of `#topo` inside the positioned wrapper `#topowrap` so `renderTopo`'s `#topo.innerHTML` rewrite doesn't erase it (`topo.js`: `runTopoFind`/`topoFindStep`); every cell has a hover copy-path button and same-name sibling rows get a distinguishing directory suffix (`tagSiblingDirs`/`distinguishingDirs`). Parent→child links are a **grey SVG connector overlay** (`paintTopo` builds `<svg class="edges">` from the visible cells' grid coords — behind the cells, `pointer-events:none`), and selecting a node greys-in its **ancestor chain (root→selection) + direct children** (`.onpath`/`.kid`; chain connectors get `.hot`), computed once per build in `computeSelPath` (`pathSet`/`childSet`, walking `parentKeyOf`). Once a centre is picked the tab grows a persistent **bar** (`#topobar`): on the **left a text filter** (`#topoFilterInput`) that prunes the tree through the *same* `buildSide` path as the type filter (matching nodes + the path back to the centre, built deep, debounced; clearing/`Esc` restores the full tree), on the **right a breadcrumb** (`#topoCrumb`) of the selection's chain **always ordered 被依賴→依賴** (sorted by signed `offsetOfKey`, so the orientation never flips), each crumb clickable to re-select, plus a **copy button** (`#topoCrumbCopy`, whole chain as newline-separated full paths via `copyCrumbChain`) and a **link button** (`#topoCrumbLink`, `copyCrumbLink` — copies a `#topo=` snapshot URL for the current centre). In **清單**, `↑`/`↓` move a keyboard cursor (`listSel`), single-click selects, double-click or `Enter` sets the centre, and switching to the tab scrolls the selected/centre row into view + flashes it. The type filter + search query persist to `localStorage` (`coir.filter`; on restore the saved types are intersected with the project's actual types so a stale type never empties the list).

**i18n** (browser only): all user-facing strings live in `src/browser/i18n.js` — a dependency-free `MESSAGES` catalog (`zh-Hant` + `en`) with `t(key, vars)` ({var} interpolation, falls back zh-Hant→key), `setLocale`/`getLocale`, and `applyStaticI18n()` which fills `index.html`'s `data-i18n` / `data-i18n-html` / `data-i18n-ph` / `data-i18n-title` attributes (those keep zh-Hant as the pre-JS fallback) plus `<title>`. `ui.js` calls `t()` for dynamic strings (`COLS` carry `labelKey`/`titleKey`); the `#langSel` switcher calls `relocalize()` (applyStaticI18n + re-render). Locale auto-detects from `navigator.language`, persisted in `localStorage.coir.lang`. Add a language by extending `MESSAGES` + `LOCALES` + an `<option>`. `src/core/` has NO user-facing strings; the **CLI is fixed English**, centralized in `src/cli.js`'s `USAGE` + `M` object (not localized).

The `?` **help modal body** is the exception to the catalog: it's authored as **Markdown** (`src/browser/help/help.{zh-Hant,en}.md`), not an HTML string in `i18n.js` — so it's easy to keep in sync. webpack imports the `.md` as a raw string (`type:'asset/source'`, no dep); `src/browser/md.js` is a ~45-line dependency-free renderer (headings/lists/bold/code/fenced/links + raw `<kbd>` passthrough, unit-tested in `test/md.test.js`) matching the `.help-body` CSS; `ui.js` `renderHelp()` renders the current locale's md into `#helpBody` on open. The import is a **dynamic `import(/* webpackMode:"eager" */ …md)`** — bundled inline (single dist file preserved) but not evaluated at module-load, so Node-run unit tests that transitively import `ui.js` don't choke on the `.md` extension (`types/index.d.ts` also has an ambient `declare module '*.md'` for tsc).

A **global type-filter bar** (`#filterbar`) sits under the banner, shared by all three tabs (one `selectedTypes` Set). It filters 清單/報告 rows to the chosen types; on 拓撲 it **prunes to branches that reach a matching type** — matching nodes plus their intermediate ancestors stay, dead branches drop, layer 0 is always kept (`buildSide` builds the full tree when filtering so deep matches keep their connecting path) — the 拓撲 bar's text filter (`topoFilterText`) folds into the *same* `buildSide` prune (type ∧ path-substring), so the two compose. The 清單 table also carries, beside the direct `被依賴`/`依賴` (in/out) degrees, two transitive-closure columns `被依賴∑`/`依賴∑` (`dependentClosure`/`dependencyClosure` sizes, computed once in `setScan`).

On first load (no project) a full-screen `#welcome` mask shows a card (pick button + brief intro); only `#langSel`, the `?` help button (`#helpBtn`), and the GitHub link are raised above the mask (`z-index:48`) so they stay clickable. `?` opens the `#help` modal (`z-index:55`: tabs / topology / search / shortcuts + a GitHub footer link) — closeable via ✕, backdrop, or Esc. Both are i18n'd; the favicon is an inline 🥥 emoji-SVG data URI. z-index ladder: welcome 45 < langSel/helpBtn/ghLink 48 < palette 50 < help 55 < usagePopup 60.

## Scope & conventions

- **3.8.x format only.** 3.5.2 is forward-compatible (tweak `meta.js`/`scan.js` if a field differs). **Not** 2.x — `.fire` scenes and old meta layout are unsupported.
- **One project at a time.**
- **Unused policy:** an asset is "unused" only if it has zero referrers **and** is **not in any bundle** (`a.bundle === 'main'`/unbundled). Any bundle's assets — `resources` AND custom Asset Bundles — are assumed to be loaded dynamically by path string at runtime, so a 0-referrer asset there is split into `unusedReport().candidates` (informational, never flagged) instead. This generalizes the old `resources/`-only rule (a prefab loaded dynamically from a custom bundle is no longer a false "unused"). Scenes are always roots, never unused.
- **Atlas utilization:** scored only for `type === 'atlas'` (sprite-atlas `.plist`) — a multi-frame png (e.g. `decal.png`, whose meta carries 2 sprite-frames) is not an atlas and is excluded. An atlas referenced as a whole `SpriteAtlas` (an edge of kind `atlas`, no `@sub`) has its frames picked by name at runtime, so per-frame usage is unknowable — these are flagged (`wholeReferenced`), not reported as 0%.
- **`dist/` is a build artifact — gitignored, not committed.** CI (`.github/workflows/deploy.yml`) runs `npm ci` → typecheck → test → build, then deploys the static site (`index.html` + `dist/` + `img/coir-topology.png` + robots/sitemap) to GitHub Pages via the **Actions** Pages source. Don't commit `dist/` or expect it in the working tree. Site images live in `img/` (README + og:image); `docs/` is markdown only.
- **No third-party runtime deps.** An earlier cytoscape/fcose graph view was removed; the bundle is now pure DOM (~60 KB). The MCP server (`src/mcp/`) is a **hand-rolled** JSON-RPC/stdio loop — deliberately no `@modelcontextprotocol/sdk` — keeping the zero-runtime-dep invariant. `package.json` dev-deps are the webpack toolchain plus `typescript`/`@types/node` (type-check only — see "Types"); the webpack entry pulls in only `src/core` + `src/browser`.
- `webpack.config.cjs` is `.cjs` (not `.js`) because `package.json` is `"type": "module"`.
- **Types (JSDoc + `.d.ts`, no `.ts` files).** The code stays `.js`; types live in `types/index.d.ts` (the data model + the `Plugin`/`PluginContext` contract) and are applied via JSDoc. `npm run typecheck` runs `tsc --noEmit` (`tsconfig.json`: `allowJs`, `checkJs:false`, `strict:false`). Checking is **opt-in per file** via `// @ts-check` — currently every non-browser file (`src/core/**`, `src/node/**`); the DOM-heavy `src/browser/**` is deliberately left unchecked. `package.json` ships `types/` (the `"types"` field) so external plugin authors get autocomplete with `/** @type {import('coir').Plugin} */`. No build/emit, no loader — `node src/cli.js`, `node --test`, and webpack are unchanged. Tighten later by flipping `checkJs:true`/`strict:true` once the browser files are annotated.
