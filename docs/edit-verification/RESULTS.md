**Phase R: 24/24 guard behaviours PASS** (R13 corrected; R13b reclassified N/A). One finding:

> **⚠ FINDING-1 — `add-component` does not validate the component type.**
> `coir edit … add-component <node> cc.Nope` returns **exit 0 and writes** a component
> entry with `__type__:"cc.Nope"`. This is inconsistent with the `set --json` path,
> which *does* reject an unknown non-`cc.` class (R13, exit 1). A typo'd type therefore
> silently produces a prefab with a phantom component. Native consequence confirmed in
> Phase N (see N17-finding: the engine cannot instantiate the bogus component).
> Note `cc.*` names are intentionally trusted as engine builtins (so coir can't know
> `cc.Nope` is fake without a builtin registry) — but a project-script typo (non-`cc.`)
> *is* knowable and still passes. Severity: low–medium (GIGO, but an inconsistent guard).

## Round 3 — fixes for FINDING-1 / FINDING-2 + native regression

Both findings fixed in the pure seam (CLI+MCP both inherit it):
- `src/edit/ops.js` add-component: resolve the type via `typeToken(scan, type)` (same resolver `set --json` uses) → reject unknown non-cc names (exit 1) AND write the **resolved** token. This also **fixes a real bug**: a project-script class name was previously written as the bare name (engine → MissingScript); it now writes the compressed uuid token.
- `src/edit/editPrefab.js` setDeep: reject an out-of-range array index (was silently null-padding); replace (`<len`) and append (`==len`) still allowed.

Unit tests added to `test/cli.test.js` (dedicated `EditAdd.prefab` fixture): unknown-type→exit1+no-write, script-class→compressed-token, OOB-set→exit2+no-null-padding, append/replace OK. **`npm test` → 158/158 pass; `tsc --noEmit` clean.**
